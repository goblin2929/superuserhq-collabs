Configuration Architecture: Cowork vs Claude Code

How instructions, skills, and team standards are structured

COWORK — 2 Layers, Flat
Layer 1 — Global (per-person)

Custom Instructions

One text box per person. Applies to all conversations.

Not enforceable — 25 people write 25 versions

Skills

ZIP uploads via Settings. Auto-triggered when relevant.

Per-person — each uploads individually
Skills and Custom Instructions are independent — no connection between them
↓
Layer 2 — Projects (shareable on Team plan)

Project: Sales BD

Instructions + uploaded docs

Shareable

Project: TDD Content

Instructions + uploaded docs

Shareable

Project: Marketing

Instructions + uploaded docs

Shareable
no connection between projects
How it works: Custom Instructions apply globally but each person writes their own. Projects are shareable but independent — Sales doesn't inherit from a "company" layer. Skills float separately, uploaded per person. No version history.
CLAUDE CODE — Multi-Layer, Shared Repo
Layer 1 — Global (loads every session)

~/.claude/CLAUDE.md

Communication style, rules, MCP integrations, memory protocol.

One version — everyone gets it

~/.claude/skills/

Global skills available in every project. prospect-audit, content-production, aeo-report...

In git repo — update once, all synced
↓ inherits
Layer 2 — Company (loads when in this repo)

novastacks/CLAUDE.md

Brand voice, service offerings, methodology, client list, content pillars. Inherited by ALL client folders below.

Company standards — always present
↓ inherits
Layer 3 — Client (loads when in this folder)

clients/gofreight/

CLAUDE.md: HubSpot portal, blog workflow, keywords, brand.

skills/: blog-publish-hubspot.md

clients/tiket/

CLAUDE.md: Internal API endpoints, product feed, brand voice.

skills/: partner-report.md

How it works: Layers stack automatically. Working on GoFreight loads Global + Company + GoFreight = 3 layers. Company methodology is always present — never copy-pasted, never forgotten. One repo, one truth. git pull to sync 25 people.
The 4 Governance Gaps at Team Scale
Why 2 layers breaks down at 25 people

1. No Enforcement

Cowork: Each person writes their own Custom Instructions. Person 1: "be formal." Person 15: blank. Person 22: "be casual." No admin control.
Code: One CLAUDE.md in the repo. Everyone gets it. VP approved it.

2. No Central Skill Management

Cowork: Each person uploads skills individually. "Everyone use /call-prep v2" — did all 25 update? No way to check.
Code: Update the skill file once. git pull. 25 people have v2. Verified.

3. No Version Control

Cowork: Someone edits the shared Project instructions. What changed? When? Why? No history. No diff. No rollback.
Code: git log shows every change — who, when, why. git revert if needed.

4. No Audit Trail

Cowork: VP asks "what skills is my team running? What changed last month?" No answer.
Code: VP opens the repo. Full visibility. Every file, every commit.
